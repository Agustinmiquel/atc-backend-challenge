## Decision y estrategia tomada

Al principio pense en varias alternativas para reducir la latencia en al respuesta del servicio. La mayoria de tecnicas que conozco son a nivel infraestructura. Decidi que para resolver el ejercicio lo mas rapido y eficiente es trabajar con la caché interna de Nest.js, el caché manager implementando Caché Aside previniendo con Fallback caché que seria como un caché de respaldo. Mejorando los tiempos de respuesta de segundos a milisegundos. A mayor escala implementaria Redis de forma distribuida para que los microservicios que necesitan la misma consistencia de datos, compartan caché. O utilizar tambien la caché que tambien se guarda en el protocolo de mensajeria como Kafka por ejemplo en la comunicacion de eventos, aunque la configuracion sea distinta. O generar varias instancias de un servicio como la API mock en la nube para liberar carga.  

Importe la dependencia al modulo haciendola global y instale una version que sea compatible con la version de Typescript para no modificar las versiones principales del proyecto. Tambien no opte por Redis para la solucion para no modificar el docker-compose y complicarme con configuraciones. Inyecte la dependencia en el handler, la instancie dentro del execute al guardarla.
Trabaje con el siguiente ejemplo:
- `http://localhost:3000/search?placeId=ChIJW9fXNZNTtpURV6VYAumGQOw&date=2022-08-25`
Combine las dos variables placeId y la date para hacer una clave unica asegurando que si aparecen los mismo datos me consolee "Cache hit". Una vez que se empieza a interar el for y a mandar peticiones en busca de la clave, se guarda ya en dos cachés a la vez el dato obtenido gracias al cacheManager. En la caché principal y la de Fallback que tienen TTL diferentes. Retornando con un manejador de errrores como Try-catch, en caso de error de la principal, la caché de respaldo y en el peor de los casos un array vacio.
